## ПРАВИЛА КОДА (это и есть подготовка к аудиту — пиши сразу так)
1. **Async-чистота.** Любой тяжёлый sync (regex, np, sqlite, file I/O, encode, Presidio) — через
   `asyncio.to_thread`/executor. Никогда не блокируй event loop. `await` на sync-клиенте loop не спасает.
2. **Fail-режим осознанный.** Security-путь: ошибка → блок (RED), fail-CLOSED. Сервисный путь
   (cache/memory): ошибка → деградация, НЕ падение пайплайна. Никогда не fail-OPEN в security
   (напр. `except: return False,[]` в матчере = дыра).
3. **ReDoS/ресурсы.** Лимит длины ввода (`max_input_length`). Никаких `[\s\S]*?` без потолка —
   лимитируй `{1,N}`. Кэш/память — с верхним пределом и eviction.
4. **Персистентность атомарна.** temp+fsync+rename. Для .npz temp-имя ОБЯЗАНО кончаться на `.npz`
   (numpy дописывает сам). JSONL не должен биться при краше посреди записи.
5. **Детерминизм/аудит.** Где есть receipt/trace_hash — он не должен зависеть от timestamp/latency
   так, чтобы ломать воспроизводимость. Версия эмбеддера зашита, mismatch → fail-safe.
6. **Untrusted input.** Всё внешнее (ingested, облако, web) — quarantine:true, через Guard.
   Метки/доверие — по хешу ТЕЛА без frontmatter, не из самого файла (anti-injection).
7. **Pydantic-валидация** на входах, нормализация (NFKC + control + zero-width + collapse) ДО хеша.

## КАК ВЫДАВАТЬ КОД (формат под передачу аудитору)
На каждый кусок:
1. **Что и зачем** — назначение, место в пайплайне, какой пункт бэклога закрывает.
2. **Контракты** — что модуль ждёт от соседей и что обещает (вход/выход, типы, исключения).
3. **Файл целиком**, запускаемый. Не псевдокод, не «...». Если длинно — бей на куски, но каждый
   кусок самодостаточен и компилируется.
4. **Self-review ПЕРЕД отдачей** (это снимает половину будущих замечаний):
   - где sync в async? где fail-OPEN? где нет лимита? где гонка/лок не тот?
   - стыки: эмбеддер верный? verdict-gating срабатывает? on_event sync?
   - `python -m py_compile` мысленно: импорты есть, синтаксис цел, HTML-эскейпов нет.
5. **Known limitations** — что осознанно НЕ сделано (с пометкой «в бэклог»), чтобы аудитор не
   тратил время на уже известное.
6. **Чек-лист для аудитора** — 3-5 конкретных пунктов «проверь особенно это» (самые рискованные места).
