## 🎯 Threat model — от кого защищаемся [РЕШЕНИЕ, новый раздел v1.3]

Без явной модели угроз выбор защиты висит в воздухе. Называю вслух.

**Профиль системы:** single-user, локальная, не подключённая к публичному API.
Я — единственный легитимный оператор.

**Противник, от которого защищаемся (в порядке вероятности):**

1. **[ОСНОВНОЙ] Вредоносный контент в RAG.** Документы из облака/веба/чужих
   источников, попадающие в ingestion — содержат prompt injection, скрытые
   инструкции, отравленный контекст. Цель противника — через документ повлиять на
   основной мозг или утечь данные. **Это 90% реальной поверхности атаки.**
1. **[ВТОРИЧНЫЙ] Инъекция через скомпрометированный плагин/синхронизацию Obsidian.**
   Запись в vault в обход моего контроля.
1. **[НИЗКИЙ] Я сам по ошибке** — забытый `train()` на сырых данных, случайный
   запуск необработанного контура.

**Из модели угроз ЯВНО ИСКЛЮЧЕНО:**

- Сетевой APT / целенаправленный взлом хоста root-уровня. Если противник получил
  root на Studio — игра окончена на уровне ОС, не security-слоя. Не моя модель.
- Инсайдер с физическим доступом (я и есть оператор).

**Следствие для дизайна:** защита оптимизируется против **контента**, не против
привилегированного процесса. Это объясняет два решения ниже:

- recall-пропуск Guard приемлем, потому что есть КАСКАД (regex + Guard + PostProcessor),
  а не один слой;
- killswitch против случайности/контента достаточен; против скомпрометированного
  процесса — отдельное OS-решение (см. инвариант 8).

## ✅ Фундамент — три модели, все dense

### 1. Основной мозг — **Qwen3.6-27B** (dense)

- **Роль** [РЕШЕНИЕ]: носитель конституции характера, диалог, код, менторство, рассуждение.
- **Происхождение** [ФАКТ, проверено]: релиз 22.04.2026, Apache 2.0, первый dense в
  семействе Qwen3.6, мультимодальный (текст/изображение/видео). Репозиторий
  `Qwen/Qwen3.6-27B`, веса на HuggingFace и ModelScope. 27.8B параметров.
- **Архитектура** [ФАКТ, проверено, ключевое для памяти]: гибрид, 64 слоя, 16
  повторяющихся блоков по схеме `3×(Gated DeltaNet → FFN) → 1×(Gated Attention → FFN)`.
  Три из четырёх подслоёв — линейное внимание (Gated DeltaNet, O(n), постоянное
  состояние вместо растущего KV). Квадратичные слои используют агрессивный GQA:
  24 query-головы / 4 KV-головы — KV cache минимизирован by design. Hidden dim 5120.
  Native context 262 144, расширяемый до ~1.01M через YaRN. Multi-Token Prediction.
- **Почему dense, не MoE** [РЕШЕНИЕ]: предсказуемость дообучения под конституцию,
  простой деплой (нет роутинга экспертов), лучшее поведение при квантизации (dense
  компрессируется предсказуемее MoE — это [ФАКТ] из обзоров деплоя, не бенчмарк).
  Оговорка [ГИПОТЕЗА]: «dense предсказуемее» верно как направление, зависит и от
  данных/кванта/промпта. Аргумент держится на FT/деплое, НЕ на бенчмарках.
- **Бенчмарки** [ФАКТ о статусе, уточнено v1.3]: SWE-bench Verified 77.2,
  SWE-bench Pro 53.5, Terminal-Bench 2.0 59.3, GPQA Diamond 87.8 — публикуются как
  реальные результаты. На Terminal-Bench 2.0 модель матчит Claude 4.5 Opus.
  В обоснование ВЫБОРА не закладываю (выбор — по FT/деплою), но для роли «основной
  мозг» это сильный сигнал: модель топовая в своём классе, не «directional».

### 2. Защита — **Qwen3Guard-Gen-4B** (dense)

- **Роль** [РЕШЕНИЕ]: Охранник (вход) + Карантин (документы/облако) + проверка вывода.
  Отдельная модель — заражение защиты не трогает основной мозг (оговорка об изоляции ниже).
- **Происхождение** [ФАКТ, проверено]: семейство Qwen3Guard (Alibaba), Apache 2.0,
  119 языков (русский в списке), размеры 0.6B/4B/8B, варианты Gen (читает полный
  prompt/response) и Stream (токенный, real-time). Обучен на 1.19M размеченных пар.
- **Метрика выбора** [ФАКТ, проверено]: workshop-paper ICLR 2026 (arXiv 2605.28830)
  «Benchmarking Open-Source Safety Guard Models». Qwen3Guard-Gen-4B даёт наивысший
  recall **83.97%**, обгоняя Llama Guard 12B и GPT-OSS Safeguard 20B (которые
  пропускают до 75% небезопасного). Recall — приоритетная метрика (пропустить
  опасное хуже ложного срабатывания).
- **⚠️ Оговорка к 83.97% [ФАКТ, важное уточнение v1.3]:** эта цифра получена в
  режиме, где **Controversial объединён с Unsafe** (strict). Мой маппинг
  `Controversial → YELLOW` (не RED) — ДРУГАЯ конфигурация, recall моей RED-категории
  будет иным. 83.97% — факт о бенчмарке, НЕ о моей системе. Замерить на своих данных.
- **Оговорка-2** [ГИПОТЕЗА, TODO замер]: высокий recall ≠ хорошая система. Не
  проверены на МОИХ данных: частота ложного quarantine, поведение на русском,
  устойчивость к обфускации, качество на коротких/длинных. Recall без замера
  false-positive — половина картины.
- **Маппинг на тройку — через нативные режимы Gen [РЕШЕНИЕ, переработано v1.3]:**
  Qwen3Guard-Gen имеет ВСТРОЕННЫЕ режимы обработки Controversial:
  **Strict** (Controversial → Unsafe) и **Loose** (Controversial → Safe).
  Это решает проблему «технические CVE-разборы тонут в карантине» без самодельной
  таблицы. Решение:
  - базовый режим — **Loose** (Controversial → YELLOW, не караню по умолчанию);
  - **Strict** точечно на категориях, где контекстный вред реален (jailbreak,
    PII-эксфильтрация);
  - маппинг: `Safe → GREEN`, `Controversial → YELLOW`, `Unsafe → RED`.
  - Контрмера-калибровка: поле `human_verdict` в security.py логирует расхождения
    мой_вердикт vs Guard — лог реальных ошибок, а не выдуманная таблица заранее.
- **Референс для defenders-learning [ФАКТ, полезное]:** вендор уже использует
  Generative Qwen3Guard как reward-сигнал в Safety RL и выпустил Qwen3-4B-SafeRL
  (дообучен на фидбеке Gen-4B). Их hybrid reward framework — готовый референс для
  моего soft-обучения защитников, не изобретать с нуля.

### 3. Атакующий — **нефильтрованная НЕ-Qwen модель 8–12B** (конкретная — TODO)

- **Роль** [РЕШЕНИЕ]: red-team генератор атак для проверки и обучения защиты.
  НЕ хакер — генератор тестовых/обучающих данных.
- **⚠️ База — НЕ Qwen [РЕШЕНИЕ, новое v1.3, критично]:** мозг (Qwen3.6) и Guard
  (Qwen3Guard на базе Qwen3) уже из одного семейства. Если атакующий тоже Qwen —
  он будет слеп ровно к тем дырам, к которым слепа защита (общая тренировочная
  слепая зона), и НЕ найдёт их. Поэтому атакующий берётся на **Mistral- или
  Llama-производной базе**. Это усиливает покрытие red-team против семейной корреляции.
- **«Нефильтрованная» — точное определение** [РЕШЕНИЕ]: отсутствие refusal-behavior
  на уровне весов при сохранении базовой связности и alignment как такового. НЕ «без
  alignment целиком» (это нестабильность). Ось — сорванный refusal, не сорванная вменяемость.
- **Свойства** [РЕШЕНИЕ]:
  - размер 8–12B (не 30B — избыточна; влезает в MLX на Air без вытеснения);
  - дообучается только на курируемых мной данных;
  - НЕ в автоцикле «найди слепые зоны → усиль» — запрещённый failure mode.
- **Два пути выбора** [TODO]:
  - Путь А — готовая аблитерированная не-Qwen модель (быстрее, мутная лицензия /
    повреждённая база — проверить).
  - Путь Б — чистая не-Qwen модель + моя расцензуровка дообучением (чище, контролирую).
- **Формализация (обязательна при активации)** [TODO, не сейчас]: до запуска
  определить классы генерируемых атак, их пропорции, критерий успеха прогона, отличие
  полезной атаки от мусора. Без этого «атакующий» — роль без критерия.
- **TODO не на критическом пути** [РЕШЕНИЕ]: фундамент защиты НЕ блокируется
  отсутствием атакующего. Защита учится на ручных/публичных датасетах (HarmBench,
  StrongREJECT, мои размеченные кейсы). Атакующий — усилитель позже, не предусловие.

## 📍 Размещение — два физических железа

### Распределение [РЕШЕНИЕ]

- **Mac Studio (64 ГБ) — ЯДРО.** Несёт: Qwen3.6-27B + Qwen3Guard-Gen-4B + RAG +
  обучающая база + конституция (точка истины). Статус: [ОЖИДАЕТСЯ, ещё не приехал].
- **MacBook Air (M5, 32 ГБ RAM, 1 ТБ) — АТАКУЮЩИЙ.** Несёт red-team модель 8–12B +
  СВОЙ отдельный Obsidian-vault атак. Статус: [В НАЛИЧИИ].

### Изоляция — аппаратная с самого начала [ФАКТ/РЕШЕНИЕ]

- Атакующий физически на другом устройстве → нет общего ядра ОС, нет общей ФС с
  RAG/обучающей базой ядра. Граница железа, не прав.
- **Единственный канал атакующий → ядро — Я** (ручной перенос только размеченного).
  Сетевого моста для обучающих данных нет.

### Сетевая сегментация [TODO, ВАЖНО]

Аппаратная изоляция ≠ сетевая. Air и Studio в одной LAN → компрометация Air может
сканировать сеть и лезть к Studio по SSH/SMB, если открыты. Меры:

- отдельный VLAN или isolated SSID для Air [РЕШЕНИЕ];
- SSH/SMB/файлошары на Studio закрыты для адреса Air [РЕШЕНИЕ];
- идеал — airgap при прогоне атак: Wi-Fi off на Air, перенос разметки через USB
  [РЕШЕНИЕ, операционная дисциплина].

### Vault атакующего — синхронизация [TODO, уточнено v1.3]

Vault атаки на Air — **локальный, без облачной синхронизации** (ни iCloud, ни
Obsidian Sync). Облачная синхронизация red-team данных = утечка наработок атак.
Перенос размеченного на Studio — только вручную через USB. [РЕШЕНИЕ]

### Несимметрия изоляции Guard [ОСТАТОЧНЫЙ РИСК, назван вслух]

Атакующий изолирован железом. Guard — только процессом на Studio, рядом с мозгом.
Но именно Guard стоит лицом к враждебному входу — самый облучаемый компонент защищён
слабее нападающего. На 64 ГБ это сознательный компромисс. Смягчение [РЕШЕНИЕ]:

- Guard в отдельном процессе с минимальными правами;
- без права записи в RAG/обучающую базу;
- перезапуск процесса после подозрительного входа.

### Размещение процессов [РЕШЕНИЕ]

|Процесс                                   |Частота            |Где               |Нагрузка       |
|------------------------------------------|-------------------|------------------|---------------|
|Свежие атаки → vault атакующего (Obsidian)|еженедельно        |Air               |лёгкая (запись)|
|Генерация атак (инференс + RAG по vault)  |по прогону         |Air               |короткая       |
|Дообучение весов атакующего               |редко (новый класс)|**Studio** (обдув)|тяжёлая, редко |
|Свежие джейлы → датасет защиты (разметка) |еженедельно        |Studio            |разметка       |

**Логика** [РЕШЕНИЕ]: свежие данные = контент в vault (дёшево, часто, инференс).
Дообучение весов = только под принципиально новый КЛАСС атак (дорого, редко).
Тяжёлое уходит на Studio с активным охлаждением — Air под троттлинг почти не попадает.

**Троттлинг Air** [ГИПОТЕЗА]: безвентиляторный корпус режет частоту при непрерывной
нагрузке >~10–15 мин (−15–25% времени). Для инференса (короткий) — неважно. Поэтому
веса атакующего дообучаются на Studio.

## 🔒 Инварианты

1. [ФАКТ] Все три модели — dense, один предсказуемый стек.
1. [РЕШЕНИЕ] Защитники сами не обучаются — только под ручным gate, изолированная база, откат.
1. [РЕШЕНИЕ] Атакующий сам не обучается — симметрично защите, человек на гейте с обеих
   сторон. Учится только на курируемых мной данных.
1. [РЕШЕНИЕ] Вывод атакующего не течёт в основной мозг и не в обучающую базу без моей разметки.
1. [РЕШЕНИЕ] Опасное (RED) → карантин + Telegram мне, не основному ИИ.
1. [РЕШЕНИЕ + названная асимметрия] Контроль над «цензурой» — через слои, которые я
   вижу и правлю (промпт + security + дообучение), не через чужую аблитерацию весов.
   **Асимметрия:** принцип держится для ЯДРА и ЗАЩИТЫ (носители точки истины).
   Атакующий сознательно выведен из-под него — Путь А (готовая аблитерация) для него
   допустим: грязный инструмент проверки, не носитель истины. Осознанное исключение.
1. [ФАКТ/РЕШЕНИЕ] Аппаратная изоляция атакующего — с самого начала (отдельный Air).
1. [РЕШЕНИЕ, уточнено v1.3] **Killswitch автообучения — двухуровневый, привилегии названы честно:**
- **Уровень софт (против случайности/контента):** read-only флаг на датасете +
  хэш-проверка обучающей базы + lock-файл перед любым `train()`. Закрывает
  «забытый debug-скрипт с trainer.train()» и отравленный контент. **ЭТО НЕ защита
  от скомпрометированного процесса** — тот же privilege level может снять флаг,
  пересчитать хэш, удалить lock. Честно: это защита от ошибки, не от злого умысла.
- **Уровень ОС (против скомпрометированного процесса) [TODO, дёшево на macOS]:**
  обучение запускается под ОТДЕЛЬНЫМ системным пользователем без прав записи на
  датасет; основной контур под этим пользователем `train()` физически не вызовет.
  Это реальный enforcement за границей привилегий. Внедрить при настройке Studio.
1. [РЕШЕНИЕ] Семейная корреляция Qwen названа: мозг и Guard из одного семейства могут
   разделять системную слепую зону. Страховка: (а) regex-слой 1 (не-нейросетевой
   механизм); (б) **атакующий на не-Qwen базе** (инвариант реализован выбором базы,
   см. модель 3); (в) периодический кросс-чек чужой моделью при разметке спорных кейсов.

## 🛡️ Доверие источника — по ingest-факту, не по пути [РЕШЕНИЕ, переработано v1.3]

**Старая дыра (v1.2):** «доверенный = мой Obsidian ядра байпасит Guard». Определение
доверия по ПУТИ файла — дыра: кто получил запись в vault (через синхронизацию, плагин),
пишет в «доверенную» папку и обходит Guard.

**Новый механизм [РЕШЕНИЕ]:** доверие НЕ по расположению, а по факту проверки.

- Документ считается доверенным, только если он **уже прошёл Guard при первичном
  ingest** и лежит в индексе с пометкой `verified_at` + хэш содержимого.
- **Любой свежий или ИЗМЕНЁННЫЙ файл всегда идёт через Guard**, независимо от папки.
  Сверка по хэшу: содержимое изменилось → пометка `verified` сбрасывается → повторный Guard.
- Компрометация синхронизации не помогает противнику: его инъекция = новое содержимое
  = новый хэш = обязательный прогон через Guard.
- «Байпас» остаётся только для уже-проверенного неизменного контента — это оптимизация
  скорости (не гонять Guard на том, что уже чисто), а не доверие по происхождению.

Следствие: нет привилегированной папки, которую достаточно скомпрометировать.

## ⚠️ Throughput-боттлнек разметки [ОСТАТОЧНЫЙ РИСК, назван вслух]

Инвариант «единственный канал — Я» делает меня горлом. Атакующий даёт 500/прогон,
я размечаю ~30/день → карантин растёт, защита учится на устаревшем срезе, появляется
соблазн «автоматизировать разметку».

**Анти-паттерн [ЗАПРЕТ]:** автоматизация разметки = прямое нарушение инварианта 4.

**Решение [РЕШЕНИЕ]:** не вся пачка подряд, а сэмплирование + приоритизация. Первым —
то, что защита ПРОПУСТИЛА (Guard дал GREEN, я при спот-проверке поймал вред): ложные
негативы опаснее ложных срабатываний. Это НЕ циклично — пропуск виден без полной
разметки, по факту расхождения GREEN vs мой спот-чек. Явный потолок N/прогон; остаток
ждёт следующего цикла или отбрасывается. Качество важнее объёма.

## 💾 Расчёт runtime-памяти Mac Studio (64 ГБ) [ФАКТ/ГИПОТЕЗА, пересчитано v1.3]

**Статичные веса** [ФАКТ, расчёт, дополнено RAG+рантайм]:

|Компонент                                          |Память                     |
|---------------------------------------------------|---------------------------|
|Qwen3.6-27B @ Q4_K_M                               |≈ 16.8 ГБ                  |
|Qwen3Guard-Gen-4B @ Q4_K_M                         |≈ 2.5 ГБ                   |
|Эмбеддер (см. ниже, e5-large/BGE-M3)               |≈ 2.2 ГБ                   |
|ChromaDB HNSW-индекс (~100K чанков, dim 1024)      |≈ 1.0–1.5 ГБ               |
|Рантайм: Ollama-сервер + ChromaDB + Python-пайплайн|≈ 1.5–2 ГБ                 |
|Резерв macOS                                       |≈ 4 ГБ                     |
|**Итого статика**                                  |**≈ 28–29 ГБ** до генерации|

(v1.2 давал 24.4 ГБ — недооценка: не учтены RAG-индекс и рантайм. Поправка ~+4 ГБ.)

**KV cache** [ГИПОТЕЗА с поправкой на архитектуру, замер на железе]:
Наивный full-attention 27B = ~256 КБ/токен → ~65 ГБ на 256K (не влезает). Но Qwen3.6
гибрид: KV копят только ~16 квадратичных подслоёв из 64, ПЛЮС эти слои используют GQA
24/4 (4 KV-головы) — двойная экономия. Реальный KV **существенно меньше ~16 ГБ на
256K** — это потолок сверху, фактически ниже. Точное число — замер реализации Gated
DeltaNet в рантайме.

**Вывод** [РЕШЕНИЕ]:

- Полный 1M контекст на 64 ГБ — недостижим. «До 1M» — теоретический максимум МОДЕЛИ,
  не возможность НА ЭТОМ ЖЕЛЕЗЕ. Из «фактов» убрано.
- Рабочий лимит контекста: **64–128K** (28–29 ГБ статика + ~8–12 ГБ KV + накладные
  укладываются). При ДВУХ параллельных инференсах (мозг + Guard одновременно) и 128K —
  ближе к грани, считать от 28–29 ГБ статики, не от 24.4. НЕ резать вслепую до 32K.
  Точный потолок — замер.

**Обучение vs инференс** [ГИПОТЕЗА, замер-blocker]:
QLoRA-дообучение 27B на Studio + одновременная работа Guard+RAG — НЕ «большой запас».
Решение: обучение и инференс НЕ одновременно (дообучение когда основной контур погашен).
Если замер покажет, что QLoRA 27B не лезет рядом с выгруженным Guard — триггер на третью машину.

## 🔤 Эмбеддер — пересмотрен под русский [РЕШЕНИЕ, новое v1.3]

**Старый выбор (v1.2):** `intfloat/multilingual-e5-base` (278M, dim 768).
**Проблема [ФАКТ]:** на русском base слаб. По MTEB: base ≈ 59.5, large ≈ 61.5,
large-instruct ≈ 64.4; на MIRACL-retrieval (ru) base = 61.6. Дельта base→large-instruct
~5 пунктов nDCG — заметно для RAG на русскоязычной конституции/Obsidian/диалоге.

**Кандидаты [РЕШЕНИЕ, выбрать на железе]:**

- **`intfloat/multilingual-e5-large-instruct`** (560M, dim 1024) — лучшая публичная
  модель <1B на массовом мультиязычном бенчмарке (обгоняет даже 7B). Прямая замена,
  тот же стек e5.
- **`BAAI/bge-m3`** (567M, dim 1024) — силён на русском, гибридный dense+sparse, что
  **ложится на мой существующий rank-bm25** (нативная гибридная retrieval вместо
  склейки двух механизмов). Если хочу dense+sparse в одной модели — это сильнее.

**Цена:** dim 768→1024 (учтено в memory-бюджете выше, ~2.2 ГБ + чуть больше ChromaDB).
Влезает легко. Рекомендация: **BGE-M3**, если ценю гибридность под bm25; иначе
**e5-large-instruct** как чистый прирост retrieval. Замерить оба на своих русских данных.

## Изменения v1.2 → v1.3

1. **Threat model назван явно** — основной противник = вредоносный контент в RAG, не
   сетевой APT. Объясняет приемлемость recall-пропуска (каскад) и достаточность soft-killswitch.
1. **Доверие источника: с пути → на ingest-факт.** Закрыта дыра «скомпрометировал папку →
   обошёл Guard». Теперь доверие по хэшу+`verified_at`, любой изменённый файл → повторный Guard.
1. **Killswitch — двухуровневый, привилегии названы честно.** Soft (против случайности) +
   OS-уровень (отдельный пользователь, против скомпрометированного процесса).
1. **Memory-бюджет пересчитан:** +RAG-индекс +рантайм → статика 28–29 ГБ, не 24.4.
   Контекст считать от неё, особенно при двух параллельных инференсах.
1. **Эмбеддер пересмотрен:** multilingual-e5-base слаб на русском → e5-large-instruct
   или BGE-M3 (+гибрид под bm25). Прирост ~5 пунктов nDCG.
1. **Guard-маппинг через нативные Strict/Loose** Gen вместо самодельной калибровки.
   Базово Loose, точечно Strict. Решает «CVE тонут в карантине» механизмом самой модели.
1. **83.97% recall — оговорка о strict-режиме.** Цифра для Controversial=Unsafe, мой
   YELLOW-маппинг даёт иное число. Факт о бенчмарке, не о системе.
1. **Атакующий — на НЕ-Qwen базе** (Mistral/Llama). Закрывает семейную корреляцию у
   корня: Qwen-атакующий был бы слеп к тем же дырам, что защита.
1. **KV cache ужесточён вниз:** учтён GQA 24/4 в квадратичных слоях — ~16 ГБ это потолок,
   реально ниже. Вилка 64–128K скорее консервативна.
1. **Бенчмарки мозга подтверждены** по источникам (SWE 77.2, Terminal-Bench матчит
   Opus 4.5) — статус поднят с «directional» до «топовая в классе».
1. **Vault атакующего — без облачной синхронизации**, перенос через USB.
1. **Cold start** — поведение при недозагрузке компонентов (fail-closed на безопасности).
1. **Латентность каскада (TTFT) и логика Stream/Gen на выходе** — вынесены в замеры-блокеры.
1. **Референс defenders-learning:** вендорский Qwen3-4B-SafeRL / hybrid reward framework
   как готовая основа soft-обучения, не изобретать.

*Примечание: «легальность лицензии» = техническая возможность свободно дообучать
(Apache 2.0 у мозга и Guard — чисто). Юридический риск обучающих данных (copyright)
Apache 2.0 НЕ снимает — это отдельный вопрос курирования датасета, не лицензии модели.*
